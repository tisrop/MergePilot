# MergeBeacon 前端设计与实现规范

本文是 MergeBeacon 前端页面、组件和样式的强制基线，适用于 `src/` 下的 Vue、TypeScript、
CSS、静态资源和前端测试。它是 [`CODE_STANDARDS.md`](CODE_STANDARDS.md) 的前端专项扩展；
`AGENTS.md` 中的产品与工程约束仍同时生效。

规则关键词：

- **必须**：合并前必须满足；违反时应阻止合并。
- **应该**：默认遵循；偏离时需在 PR 中说明原因和替代保障。
- **禁止**：不得以临时样式、静默降级或关闭检查绕过。

规范描述的是当前已实现的设计语言，不用于声明尚未实现的主题、平台能力或交互。若实现与本文不一致，
必须先确认是实现偏差还是设计系统需要演进；修改全局基线时必须同步更新本文。

## 1. 产品与视觉原则

1. **专业桌面工作台**：界面服务于代码合并、评审和问题处理，优先保证信息层级、可扫描性和操作
   确定性，不追求营销页式装饰。
2. **紧凑但不拥挤**：保持适合开发者工具的信息密度；相关信息就近分组，主要操作突出，次要信息
   降低视觉权重。
3. **深海军蓝 + 薄荷绿品牌语言**：海军蓝表达稳定和主操作，薄荷绿用于品牌识别与点缀。薄荷绿
   不得替代成功、Diff 新增等语义色。
4. **内容优先**：阴影、动画、渐变只能辅助层级或反馈。禁止大面积装饰渐变、持续运动、玻璃拟态、
   霓虹效果或与代码评审无关的视觉噪声。
5. **状态可辨识**：加载、空数据、失败、禁用、选中、危险和部分成功必须有明确且一致的反馈，不能
   只依赖颜色区分。
6. **跨平台一致**：macOS、Windows、Linux 上应保持相同的信息架构与操作语义，同时允许系统字体、
   字体渲染和原生窗口行为存在合理差异。

## 2. 前端架构边界

- 使用 Vue 3 Composition API 和 `<script setup lang="ts">`；不得新建 Options API 组件。
- 页面组件负责路由级编排，领域组件负责一组相关业务交互，共享组件负责无业务归属的通用能力。
- 跨页面业务状态必须放入 Pinia。组件不得复制 store 已承担的请求、缓存或竞态控制逻辑。
- `src/api/index.ts` 是唯一 Tauri IPC 入口；组件和 store 禁止直接调用 `invoke` 或其他 Tauri
  command API。
- 远端数据、平台能力、异步生命周期和安全规则必须遵循 `CODE_STANDARDS.md`，视觉优化不得改变
  这些语义或以占位数据掩盖缺失能力。
- 未经讨论禁止引入 UI 框架、Tailwind、CSS-in-JS、图标字体或第二套状态管理方案。

## 3. 设计 Token

### 3.1 Token 使用原则

- 全局 Token 统一定义在 `src/style.css` 的 `:root` 中。组件必须优先使用已有 Token，禁止复制
  已有颜色、间距、圆角、阴影或动效数值。
- 只有跨页面复用且语义稳定的值才能提升为全局 Token；单个组件的局部尺寸应保留在组件作用域内。
- Token 必须按语义命名，不使用 `blue-500`、`space-small-2` 等依赖具体外观或顺序的名称。
- 修改现有 Token 时必须检查全部使用点，确认不会破坏 Diff、严重级别、焦点或平台状态语义。

### 3.2 颜色

当前颜色基线如下，实际值以 `src/style.css` 为准：

| 用途 | Token | 约束 |
| --- | --- | --- |
| 页面背景 | `--color-bg`、`--color-bg-accent` | 用于应用背景和弱分区 |
| 表面 | `--color-surface`、`--color-surface-raised` | 卡片、面板和浮层 |
| 边框 | `--color-border`、`--color-border-light` | 普通和弱分隔 |
| 正文 | `--color-text` | 标题和主要内容 |
| 次要文字 | `--color-text-secondary`、`--color-text-tertiary` | 辅助信息和弱提示 |
| 主操作 | `--color-primary*` | 导航、链接、主要按钮和选中状态 |
| 品牌点缀 | `--color-brand-accent*` | BrandMark 和克制的品牌强调 |
| 焦点 | `--color-focus` | 键盘焦点，不得被品牌色替代 |
| 语义状态 | `--color-success*`、`--color-warning*`、`--color-danger*` | 成功、警告、危险 |
| Diff | `--diff-add-*`、`--diff-remove-*` | 新增和删除代码，语义不可互换 |
| AI 严重级别 | `--severity-critical/major/minor/info` | 评审问题等级 |

要求：

- 文本和关键控件必须保持清晰对比度；弱化信息不得弱到影响阅读。
- 危险操作必须同时使用文字或图标表达，不得只显示红色。
- 品牌薄荷绿、成功绿、Diff 新增绿虽然接近，但必须保持不同的 Token 和用途。
- 新增颜色前必须证明现有语义色无法表达；禁止在多个组件中散落相近的十六进制颜色。

### 3.3 字体与排版

- 正文使用 `--font-sans`；代码、Diff、提交哈希、行号和技术标识使用 `--font-mono`。
- 基础字号为 `14px`，基础行高为 `1.5`。正文不得小于 `12px`，主要交互文字通常不得小于
  `13px`。
- 标题层级必须通过字号、字重和间距共同表达。一个页面只应有一个视觉主标题。
- 按钮和标签可使用 `600` 字重；长正文避免全段粗体。
- 远端标题、仓库名、分支名、模型 ID 和错误正文必须支持安全换行、截断或省略，不能撑破布局；
  被省略的关键值应该提供可访问的完整信息，例如 `title` 或邻近详情区域。

### 3.4 间距、圆角与阴影

- 使用基于 `4px` 的 `--space-1/2/3/4/5/6/8/10/12` 间距阶梯。
- 同组元素使用较小间距，组与组之间使用较大间距；禁止为对齐问题连续添加任意像素补丁。
- 使用 `--radius-sm/md/lg/xl`。表单和按钮通常使用 `md`，卡片通常使用 `lg`，品牌或大型容器
  才使用 `xl`。
- 使用 `--shadow-sm/md/lg/xl` 表达层级。常规卡片保持轻阴影，浮层应明显高于页面内容；禁止用
  重阴影代替边框或信息分组。

### 3.5 动效

- 使用 `--transition-fast/base/slow`，并明确列出过渡属性；禁止 `transition: all`。
- 动效只用于状态变化、浮层出现和操作反馈，通常控制在 `150–300ms`。
- 悬停位移应克制，不能导致布局抖动或误触。
- 必须保留 `prefers-reduced-motion: reduce` 支持；核心信息和操作不得依赖动画才能被理解。

## 4. 布局与响应式

- 应用采用桌面优先布局，默认窗口为 `1280 × 800`，最小支持窗口为 `900 × 600`；新页面必须在
  这两种尺寸下可用。
- 全局布局使用 `--sidebar-width: 264px` 和 `--header-height: 64px`。修改这两个值属于设计系统
  变更，必须检查侧边栏、页头、滚动容器和窄窗口行为。
- 页面内容区应沿用布局容器提供的内边距和滚动边界。禁止页面自行创建不必要的全窗口滚动层。
- 在 `900px` 临界宽度附近必须检查导航、页头和主体内容；窄布局可以压缩间距或重排，但不得
  隐藏关键操作、状态或错误信息。
- 页面不得出现非预期的整体横向滚动。Diff、长路径和代码内容可在其专属容器内横向滚动。
- 固定、粘性和绝对定位只能用于确有层级需求的导航、浮层或锚点；必须验证滚动、缩放和最小窗口。
- 内容加载前后应尽量保持布局稳定，避免按钮、标题和列表发生无必要跳动。

## 5. 组件分层与复用

### 5.1 放置规则

- 路由级页面放在 `src/pages/`。
- 具备明确领域语义的组件放在 `src/components/ai/`、`diff/`、`issue/`、`pr/`、`review/`
  等对应目录。
- 布局和导航放在 `src/components/layout/`。
- 无领域依赖、可跨页面复用的组件放在 `src/components/shared/`。
- 组件出现第二个真实复用点，或复制内容已包含状态、键盘交互和可访问性逻辑时，应该提取共享组件。

### 5.2 现有基础能力

新增界面必须先复用或扩展以下能力，不得创建外观相同但行为不一致的副本：

- 全局类：`.card`、`.card-hover`；
- 按钮类：`.btn`、`.btn-primary`、`.btn-success`、`.btn-danger`、`.btn-reopen`、`.btn-sm`；
- 表单类：`.input`；
- 浮层类：`.dropdown-panel`、`.dropdown-option`；
- 状态与工具类：`.badge-*`、`.skeleton`、`.text-secondary`、`.font-mono`；
- 共享组件：`src/components/shared/AppSelect.vue`、`BrandMark.vue`。

扩展共享基础能力时，必须保持原有 API 和状态兼容，或同步迁移全部使用点并增加相应测试。

## 6. 组件规范

### 6.1 按钮与操作

- 每个操作区只保留一个明确主操作；次要操作使用普通或文本型视觉层级。
- 按钮文案必须使用动词并表达结果，例如“开始评审”“重新加载”“合并 PR”，避免“确定”“提交”等
  缺乏上下文的词。
- 危险或不可逆操作使用危险样式，并在必要时二次确认；确认信息必须包含目标仓库、PR/Issue 或
  将产生的后果。
- 异步提交期间必须防止重复触发并显示进行中状态；失败后恢复可操作状态，取消不显示为错误。
- 禁用按钮必须同时具有原生 `disabled` 语义和清晰外观，不能只通过降低透明度模拟禁用。
- 常规交互控件高度应保持约 `36px` 或更大；紧凑控件不得牺牲键盘可达性和可辨识度。

### 6.2 表单

- 每个输入必须有可访问的标签；占位符只用于示例或格式提示，不能替代标签。
- 必填、格式、范围和平台能力应在提交前或提交时给出就近提示；错误提示必须说明如何修复。
- 表单错误不能只用颜色表达，应包含文字和相应状态语义。
- 密钥和 Token 默认遮蔽，禁止写入日志、DOM 调试输出或持久化到非安全位置。
- 自定义选择器优先使用 `AppSelect`；新增自定义控件必须实现方向键、Enter/Space、Escape、焦点
  管理和外部点击关闭，并在卸载时移除监听器。

### 6.3 卡片、列表与表格

- 卡片用于表达独立内容组，不得为每层容器都嵌套卡片和阴影。
- 可点击卡片必须使用真实链接或按钮语义，提供 hover、active 和 focus-visible 状态。
- 列表项应稳定对齐主要字段和操作，避免不同数据长度造成操作列漂移。
- 数据量较大时优先使用分页、增量加载或虚拟化策略；不得一次渲染无边界远端数据。
- 表格必须有可理解的表头；小窗口下应优先保留关键列，次要信息可重排到行内详情。

### 6.4 浮层、菜单与提示

- 浮层必须有明确层级、边界和关闭方式；Escape 和点击外部应按控件语义关闭。
- 浮层不得被父级 `overflow` 意外裁剪，也不得遮挡当前操作所需的关键上下文。
- 仅靠 hover 出现的内容不能承载完成任务所必需的信息；关键提示必须支持键盘焦点。
- 通知和错误应就近展示。只有跨页面或后台操作才使用全局通知，禁止堆叠重复提示。

### 6.5 加载、空状态与错误

- 首次加载可使用 skeleton 或明确加载提示；局部刷新应尽量保留已有内容。
- 空状态必须区分“确实无数据”和“尚未选择仓库/平台”，并提供下一步操作。
- 可重试失败必须保留已成功加载的数据并提供重试入口。
- 不支持的平台能力必须明确说明原因，禁止隐藏按钮后让用户误以为功能不存在或静默降级。
- 部分成功必须同时展示已完成事实与后续失败，例如“PR 已合并，但关联 Issue 关闭失败”。

## 7. 代码、Diff 与 AI 评审界面

- 代码、行号、Diff 和提交哈希必须使用等宽字体并保持可复制性。
- Diff 新增和删除必须分别使用 `--diff-add-*` 与 `--diff-remove-*`；不得反转、复用品牌色或仅靠
  背景色区分，行前标记和文本语义也应保留。
- 超长代码行应在 Diff 容器内部滚动，不得导致整个页面横向滚动。
- AI 严重级别必须使用 `critical`、`major`、`minor`、`info` 的既有语义映射，不能因视觉偏好
  调换颜色或名称。
- 远端评论、标题、模型 ID 和错误正文一律作为纯文本渲染；禁止使用 `v-html`。
- `DiffViewer.vue` 使用 diff2html 将纯 Diff 转换为本地展示结构，是当前唯一受控的 `v-html`
  例外；不得把该例外扩展到评论、Markdown 或任意远端 HTML。
- Markdown 或富文本能力若未来需要引入，必须先完成可信解析、HTML 清洗、链接策略与 CSP 评审。

## 8. 交互、状态与生命周期

- 所有可交互元素必须覆盖默认、hover、active、focus-visible、disabled，以及适用的 loading、
  selected 和 error 状态。
- 视觉选中状态必须与真实路由或业务状态一致，不得只维护组件内部的临时高亮。
- 请求发起后，页面必须让用户知道正在处理什么；请求结束后必须清理 loading，无论成功、失败还是
  取消。
- 组件添加的事件监听器、计时器、观察器和流式监听必须在卸载时清理。
- 异步响应写入状态前必须验证平台、仓库、序列号或 `request_id`；视觉优化不得移除这些竞态保护。
- 键盘激活与指针激活必须产生相同业务结果；禁止把核心操作仅绑定到 `mouseenter`、双击或手势。

## 9. 可访问性

- 优先使用原生语义元素：操作使用 `button`，导航使用 `a`/`RouterLink`，输入使用对应表单元素。
  禁止用无语义 `div` 模拟按钮，除非补齐角色、键盘和状态语义且确有必要。
- 必须保留全局 `:focus-visible` 轮廓和布局中的跳转链接。组件不得用 `outline: none` 移除焦点提示。
- 图标按钮必须有可访问名称；纯装饰图标应从辅助技术中隐藏。
- 导航、选择器、展开控件和异步区域应使用恰当的 `aria-label`、`aria-current`、`aria-pressed`、
  `aria-expanded`、`aria-live` 等属性，但禁止用 ARIA 替代原生语义。
- 不得只用颜色、位置或动画传达状态；必须同时提供文字、符号或结构信息。
- 文本缩放、系统字体差异和最小窗口下，核心内容不得被裁切或覆盖。
- 新增动效必须验证 reduced-motion；自动聚焦和焦点转移必须服务于明确操作，不能打断阅读。

## 10. 文案与图标

- 用户可见文案使用简体中文；平台、协议、代码标识和官方产品名保留其标准英文写法。
- 文案应简洁、具体、可执行：说明发生了什么、影响什么、用户下一步能做什么。
- 相同动作必须使用相同术语，例如“拉取请求（PR）”“合并”“评审”“仓库”，不得在同一流程中
  随意切换近义词。
- 图标必须辅助而非替代关键文案。只有行业普遍理解且有可访问名称时，操作才可仅显示图标。
- 品牌标记统一复用 `BrandMark.vue` 及应用图标的“分支汇合后向右导航”语言；禁止在页面中手绘
  近似但比例、颜色或语义不同的品牌图形。
- Emoji 不作为正式功能图标；确需表达轻量情绪时应确保跨平台差异不会影响含义。

## 11. CSS 实现规则

- 组件样式默认使用 `<style scoped>`；只有 Token、reset、共享基础类和跨页面工具类放入
  `src/style.css`。
- 选择器保持扁平，优先使用单一类名；禁止依赖脆弱的深层 DOM 结构或过高特异性。
- 禁止使用 `!important`，全局 reduced-motion 覆盖等明确的无障碍兜底除外。
- 禁止 `transition: all`；禁止以大量任意 `z-index` 竞争层级。新增全局层级前应定义清晰用途。
- 若已有 Token，不得硬编码重复值。局部颜色也必须先确认不是全局语义状态。
- CSS 必须处理长文本、空内容、不同语言长度、系统字体和缩放，不得只按演示数据调布局。
- 浏览器实验属性必须提供可接受降级；桌面 WebView 差异不得导致关键内容或操作不可用。
- 删除组件或状态时必须同步清理失效样式，禁止长期保留不可达选择器。

## 12. 测试与视觉验收

### 12.1 必须增加测试的变化

以下变化除实现外必须增加或更新 Vitest + Vue Test Utils 测试：

- 共享组件 API、键盘交互、展开/关闭和焦点行为；
- loading、空状态、错误、重试、禁用、部分成功和平台不支持状态；
- 条件显示的危险操作或平台能力；
- 会影响平台、仓库、分页、请求序列号或 AI `request_id` 的交互；
- 修复可访问性、监听器泄漏或异步竞态缺陷。

纯 Token 或静态 CSS 调整可不增加单元测试，但必须完成视觉验收。

### 12.2 视觉验收清单

修改页面、组件或全局样式后，至少检查：

- 默认窗口 `1280 × 800` 与最小窗口 `900 × 600`；
- 首次加载、已有数据刷新、空数据、失败、重试和禁用状态；
- 长仓库名、分支名、模型 ID、中文错误和超长代码行；
- 鼠标 hover/active 与键盘 Tab、Enter/Space、Escape；
- focus-visible、文本缩放和 reduced-motion；
- 页面级横向溢出、浮层裁切、滚动容器冲突和内容跳动；
- 深海军蓝、品牌薄荷绿、成功绿、Diff 色和 AI 严重级别没有语义串用；
- macOS、Windows、Linux 中至少对受影响的平台差异做静态审查；涉及原生窗口或系统字体时应在
  对应平台实测。

### 12.3 提交门禁

按变更范围运行检查。`npm run check:frontend-standards` 是 GitHub CI 的阻断项，用于检查
Tauri IPC 边界、`v-html`、未批准 UI 框架、焦点轮廓、显式 transition 和 `!important` 等可静态
识别的硬规则。前端代码或样式变更至少执行：

```bash
npm run lint
npm run check:frontend-standards
npm run format
npm run build
npm test
```

仅修改 Markdown 规范时至少执行 `git diff --check`。提交前还必须确认没有无关生成物、临时截图、
本地工具状态或全仓格式化进入变更。
